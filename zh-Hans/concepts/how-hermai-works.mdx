## 两端分工

Hermai 有两端:

1. **CLI**([hermai-cli](https://github.com/hermai-ai/hermai-cli),开源)— 在你的机器上运行。贡献者用它检视网站、编写描述端点的 schema,然后把 schema 推送到 registry。

2. **Platform**(api.hermai.ai + fetch.hermai.ai)— schema registry **加上**托管执行接口。Registry 存储并提供贡献者推送的 schema;agent 在运行时查询以获取 registry 中任何域名的即时、结构化端点数据。之后 agent 可以选择用 catalog 数据直接调用上游网站,或让 Hermai 通过 [cloud fetch](/zh-Hans/concepts/cloud-fetch) 代为执行。

```
Contributor                        Platform                         Agent
    │                                  │                               │
    │  discovery toolkit (local)       │                               │
    │  hermai registry push            │                               │
    │──────────────────────────────>   │                               │
    │                                  │   GET /v1/catalog/{domain}    │
    │                                  │ <────────────────────────────│
    │                                  │   { endpoints, actions }      │
    │                                  │ ────────────────────────────> │
    │                                  │                               │
    │                                  │   选项 A:agent 自己调用       │
    │                                  │                        curl ──┘
    │                                  │                               │
    │                                  │   选项 B:POST /v1/fetch       │
    │                                  │ <────────────────────────────│
    │                                  │  (Hermai 代为调用、处理        │
    │                                  │   anti-bot、回干净 JSON)      │
    │                                  │ ────────────────────────────> │
```

## CLI 做什么

CLI 提供一组确定性的工具组 — 每个子命令做一件事,并输出可供下一步处理的 JSON:

- **`hermai detect`** — 分类网站平台(Shopify、WordPress、Next.js 等)并检测 anti-bot 系统。
- **`hermai wellknown`** — 探测 `robots.txt`、sitemap、RSS feed 与 GraphQL 端点。
- **`hermai probe --body | hermai extract`** — 从页面提取嵌入数据。可识别 13 种 SSR/hydration 模式(`__NEXT_DATA__`、`ytInitialData`、`__APOLLO_STATE__`、`SIGI_STATE` 等)。
- **`hermai intercept`** — 启动浏览器并捕获真实 XHR 流量,适用于动态功能(搜索、购物车、筛选)。
- **`hermai introspect`** — 获取 GraphQL schema。
- **`hermai session bootstrap`** — 为 anti-bot 网站预热浏览器 session。

贡献者根据发现结果编写 schema JSON 文件,并通过 `hermai registry push` 推送。不需要 LLM key 或托管服务。

## Platform 做什么

Platform 有两个接口:

**Catalog(只读、免费):**

- **`GET /v1/catalog/{domain}`** — 返回该域名所有端点与 action。主要的 agent 查询接口,需要 [intent](/zh-Hans/concepts/catalog#intent) 说明用途。
- **`GET /v1/schemas`** — 浏览并搜索整个 registry。
- **`POST /v1/schemas`** — 推送新 schema。将按照 [intent 分类](/zh-Hans/concepts/catalog#intent-taxonomy) 进行校验并立即发布。

**Cloud fetch(托管执行、按次计费):**

- **`POST https://api.hermai.ai/v1/fetch`** 配 `{"url":"…"}` — Hermai 代你调用该 URL,并按匹配到的 schema 投射响应。最适合本来就吃纯 HTTPS 的公开 API。
- **`POST https://fetch.hermai.ai/v1/fetch`** 配 `{"site":"…","endpoint":"…","params":{…}}` — Hermai 处理 anti-bot、session、签名请求等硬站需求(X/Twitter、Booking、Threads、Instagram、TikTok 等)。同样用 Bearer 密钥认证。

Cloud fetch 可选 — 对没有 anti-bot 的网站,agent 也可以拿 catalog 数据自己调用。完整参考见 [Cloud fetch](/zh-Hans/concepts/cloud-fetch)。

## 为什么要这样分工?

发现很贵(浏览器 + LLM + 时间)。查询很便宜(数据库读取)。执行则介于两者之间,视网站而定。分开之后:

- **贡献者**只需在自己机器上,本地付一次发现成本
- **之后的每个 agent**都能免费获得即时结构化 catalog 数据
- **只想拿干净 JSON 的 agent**调用 cloud fetch,免去自己写 HTTP / proxy / cookie / 签名代码
- **Platform**各层独立扩展 — registry、catalog、execution 是分开的服务
- **不锁死平台** — CLI 是开源(AGPL-3.0)且可离线运行;Cloud fetch 按次选用
