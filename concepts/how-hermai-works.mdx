## The split

Hermai has two sides:

1. **The CLI** ([hermai-cli](https://github.com/hermai-ai/hermai-cli), open source) — runs on your machine. Contributors use it to inspect a site and compose a schema describing its endpoints, then push the schema to the registry.

2. **The platform** (api.hermai.ai + fetch.hermai.ai) — a schema registry **plus** a hosted execution surface. The registry stores and serves the schemas that contributors have pushed; agents query it at runtime to get instant, structured endpoint data for any domain in the registry. Agents can then either call the upstream site directly with the catalog data, or have Hermai run the call for them through [cloud fetch](/concepts/cloud-fetch).

```
Contributor                        Platform                         Agent
    │                                  │                               │
    │  discovery toolkit (local)       │                               │
    │  hermai registry push            │                               │
    │──────────────────────────────>   │                               │
    │                                  │   GET /v1/catalog/{domain}    │
    │                                  │ <────────────────────────────│
    │                                  │   { endpoints, actions }      │
    │                                  │ ────────────────────────────> │
    │                                  │                               │
    │                                  │   Option A: call site direct  │
    │                                  │                        curl ──┘
    │                                  │                               │
    │                                  │   Option B: POST /v1/fetch    │
    │                                  │ <────────────────────────────│
    │                                  │   (Hermai runs the call,      │
    │                                  │    handles anti-bot, returns  │
    │                                  │    clean JSON)                │
    │                                  │ ────────────────────────────> │
```

## What the CLI does

The CLI ships a deterministic toolkit — each subcommand does one thing and emits JSON the next step can consume:

- **`hermai detect`** — classify the site's platform (Shopify, WordPress, Next.js, etc.) and detect anti-bot systems.
- **`hermai wellknown`** — probe `robots.txt`, sitemaps, RSS feeds, and GraphQL endpoints.
- **`hermai probe --body | hermai extract`** — pull embedded data from a page. Recognises 13 SSR/hydration patterns (`__NEXT_DATA__`, `ytInitialData`, `__APOLLO_STATE__`, `SIGI_STATE`, and more).
- **`hermai intercept`** — launch a browser and capture real XHR traffic, useful for dynamic features (search, cart, filters).
- **`hermai introspect`** — fetch a GraphQL schema.
- **`hermai session bootstrap`** — warm a browser session for anti-bot sites.

The contributor composes a schema JSON file from what they find and pushes it with `hermai registry push`. No LLM key or managed service is required.

## What the platform does

The platform has two surfaces:

**Catalog (read-only, free):**

- **`GET /v1/catalog/{domain}`** — returns all endpoints and actions for a domain. The main agent-facing lookup. Requires an [intent](/concepts/catalog#intent) describing why you need it.
- **`GET /v1/schemas`** — browse and search the full registry.
- **`POST /v1/schemas`** — push a new schema. Validated against the [intent taxonomy](/concepts/catalog#intent-taxonomy) and published immediately.

**Cloud fetch (hosted execution, billed per call):**

- **`POST https://api.hermai.ai/v1/fetch`** with `{"url":"…"}` — Hermai calls the URL for you and projects the response through the matched schema. Right for public APIs that already work over plain HTTPS.
- **`POST https://fetch.hermai.ai/v1/fetch`** with `{"site":"…","endpoint":"…","params":{…}}` — Hermai handles anti-bot, sessions, and signed requests for hard sites (X/Twitter, Booking, Threads, Instagram, TikTok, etc.). Same Bearer-key auth.

Cloud fetch is optional — for sites without anti-bot protection, your agent can still call the upstream directly using the catalog data. See [Cloud fetch](/concepts/cloud-fetch) for the full reference.

## Why this split?

Discovery is expensive (browser + LLM + time). Lookup is cheap (database read). Execution is somewhere in between depending on the site. By separating them:

- **Contributors** pay the discovery cost once, locally, on their own machine
- **Every agent after them** gets instant structured catalog data, free
- **Agents that just want clean JSON** call cloud fetch and skip building HTTP / proxy / cookie / signer code themselves
- **The platform** scales each layer independently — registry, catalog, and execution are separate services
- **No vendor lock-in** — the CLI is open source (AGPL-3.0) and works offline; cloud fetch is opt-in per call
